## 1. Problem Statement

The Consent & Preferences Management System (CPMS) currently lacks first-class support for standardized written and spoken language codes on preference records. The legacy Enterprise Preference Management System (EPMS) uses a homegrown language code table; CPMS must adopt the ISO 639-3 standard while preserving backward compatibility with the EPMS code surface.

This whitepaper defines the solution architecture for:

1. A new **Language Code Reference Data Service** that manages ISO 639-3 codes, custom code sources (including the EPMS homegrown codes), versioned snapshots, and bidirectional code mappings.
2. **Preference schema extensions** to carry both ISO 639-3 codes (authoritative) and legacy EPMS codes (preserved for backward compatibility).
3. A **phased migration strategy** that safely transforms EPMS language codes on existing preferences to ISO 639-3.
4. **Legacy EPMS API compatibility** via pass-through proxy services that translate between EPMS and CPMS formats.
5. An **Admin UI control plane** for viewing and managing language codes and mappings.

## 2. Background & Context

| System | Role | Language Code Approach |
|--------|------|------------------------|
| **CPMS** | Modern cloud-first consent & preference platform | Moving to ISO 639-3 (this work) |
| **EPMS** | Legacy system being replaced | Homegrown language code table |

**ISO 639-3** is the international standard for language codes maintained by SIL International as the Registration Authority. It covers ~7,916 languages using 3-letter alpha codes (e.g., `eng` for English, `spa` for Spanish). SIL publishes machine-readable data files and issues annual updates. HL7 FHIR already uses ISO 639-3 as a canonical terminology system — relevant precedent for healthcare compliance contexts.

**EPMS language codes** are a proprietary code table maintained by Humana. They encode written and spoken language preferences using a homegrown scheme. These codes are not mappable 1:1 to ISO 639-3 without an explicit translation table.

**Versioning approach (confirmed):** Versioning means *snapshot versioning* — dated imports of code sets — plus *deprecation flags* on individual codes. CPMS does not need to maintain multiple coexisting active versions of the same code identifier. Codes are never deleted; only deprecated with a `deprecated_date` and optional `replaced_by` reference.

## 3. Goals & Non-Goals

### Goals
- Adopt ISO 639-3 as the canonical language code standard in CPMS preference records.
- Introduce a dedicated Language Code Reference Data Service as the single source of truth for all language codes and cross-system mappings.
- Extend the CPMS preference schema to carry both ISO 639-3 codes and legacy EPMS codes, with no breaking changes to existing consumers.
- Support bidirectional transformation between EPMS language codes and ISO 639-3 codes.
- Preserve all legacy EPMS API interfaces via pass-through proxy with transformation.
- Provide an Admin UI control plane (read-only initially; CRUD in a future phase).
- Support additional code sources beyond ISO 639-3 and EPMS without schema changes.

### Non-Goals
- Automated bulk import without human review (all code entry requires human sign-off, per compliance requirements).
- Real-time language code governance or workflow (future phase).
- Graph-based multi-hop translation (e.g., EPMS → HL7 FHIR via ISO 639-3 intermediate); direct mappings only in this phase.
- Retirement or sunset of legacy EPMS interfaces (no sunset plan at this time).

### 5.2 CPMS Preference Schema Extension

The existing CPMS preference record will be extended with new language code fields. Existing fields (`written_language_code`, `spoken_language_code`) will transition from carrying EPMS codes to carrying ISO 639-3 codes.

**New fields added to the preference record:**

| Field | Type | Description |
|-------|------|-------------|
| `written_language_iso_code` | VARCHAR(10) | ISO 639-3 alpha-3 code (e.g., `eng`). This becomes the authoritative written language field. |
| `written_language_legacy_epms_code` | VARCHAR(50) | Legacy EPMS homegrown code. Preserved for backward compatibility. Populated during migration and maintained by the legacy proxy layer. |
| `spoken_language_iso_code` | VARCHAR(10) | ISO 639-3 alpha-3 code (e.g., `spa`). Authoritative spoken language field. |
| `spoken_language_legacy_epms_code` | VARCHAR(50) | Legacy EPMS homegrown code for spoken language. |

**Migration note on existing fields:** The existing `written_language_code` and `spoken_language_code` fields will be repurposed to hold ISO 639-3 codes post-migration (they previously held EPMS codes). This repurposing is safe because the migration is the transition point — all consumers must be informed that these fields now carry ISO 639-3 values.

**CPMS API, Kafka messages, and file outputs** will expose both ISO codes and legacy EPMS codes in their payloads. The ISO fields are the canonical representation; the legacy fields are additive and backward-compatible.

## 6. Language Code Service — API Design

Base path: `/api/v1/language-codes`

### 6.1 Code Sources

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/code-sources` | List all code sources |
| `GET` | `/code-sources/{id}` | Get a specific code source |
| `GET` | `/code-sources/{id}/snapshots` | List all snapshots for a source |
| `GET` | `/code-sources/{id}/snapshots/latest` | Get the active snapshot |

### 6.2 Language Codes

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/codes` | List codes. Query params: `source_id`, `status`, `search`, `supports_written`, `supports_spoken`, `page`, `size` |
| `GET` | `/codes/{id}` | Get a specific code by UUID |
| `GET` | `/codes/lookup?source={sourceName}&code={code}` | Look up by source name + code value (the common consumer call) |
| `POST` | `/codes` | Create a new code (admin; triggers `language_code.created` Kafka event) |
| `PATCH` | `/codes/{id}` | Update code metadata (admin) |
| `POST` | `/codes/{id}/deprecate` | Deprecate a code, optionally specifying a replacement (admin) |

### 6.3 Code Mappings

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/mappings` | List mappings. Query params: `source_code_id`, `source_name`, `target_name`, `status` |
| `GET` | `/mappings/{id}` | Get a specific mapping |
| `POST` | `/mappings` | Create a mapping (admin) |
| `PATCH` | `/mappings/{id}` | Update a mapping (admin) |
| `POST` | `/mappings/{id}/deprecate` | Deprecate a mapping (admin) |

### 6.4 Translation (Core Consumer API)

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/translate` | Translate a single code between systems |
| `POST` | `/translate/batch` | Translate multiple codes in one request (for migration jobs) |

**Single translation request:**
```json
{
  "code": "HUM-ENG-01",
  "from_source": "EPMS",
  "to_source": "ISO_639_3"
}
```

**Single translation response:**
```json
{
  "original_code": "HUM-ENG-01",
  "translated_code": "eng",
  "from_source": "EPMS",
  "to_source": "ISO_639_3",
  "mapping_id": "uuid",
  "mapping_type": "equivalent",
  "confidence": "exact",
  "status": "active"
}
```

**Batch translation** accepts an array of the same request shape and returns an array of responses, including partial failures (a failed translation returns `error` instead of `translated_code`).

### 6.5 Import Management

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/imports` | Submit an import job for review. Body: `{ code_source_id, payload_location, snapshot_version, notes }` |
| `GET` | `/imports/{id}` | Get import job status |
| `POST` | `/imports/{id}/approve` | Approve and apply the import (admin) |
| `POST` | `/imports/{id}/reject` | Reject the import with notes (admin) |

### 7.3 Consumer Caching Pattern

Each consuming service (Preference Service, EPMS Compatibility Layer, Migration Job) maintains a local in-memory or distributed (Redis) cache of language codes and mappings:

1. **On service startup:** Call `GET /codes` and `GET /mappings` from the Language Code Service to seed the local cache. Paginate until all records are loaded.
2. **Ongoing:** Subscribe to `cpms.language-codes.*` Kafka topics. On each event, update the local cache entry (add, update, or mark deprecated).
3. **Cache key design:** `(source_name, code_value)` → `language_code` record; `(source_name, code_value, target_source_name)` → `code_mapping` record.
4. **Cache TTL fallback:** 24-hour TTL as a safety net if Kafka delivery is disrupted. Services re-seed from REST API at TTL expiry.
5. **Lookup failures:** If a code is not found in cache (unmapped), the consuming service must decide per business rules: use the original code as-is, reject, or flag for manual review.

## 8. Migration Strategy

Migration is executed in four sequential phases. Phases 1 and 2 may overlap; Phases 3 and 4 are strictly sequential.

### Phase 1: EPMS Preference Migration (Pre-existing work, prerequisite)

- Migrate all existing preferences from EPMS to CPMS.
- During migration, populate `written_language_code` and `spoken_language_code` fields on CPMS preferences using EPMS language codes (as-is, no transformation yet).
- **Outcome:** CPMS has all preferences with EPMS language codes in the language code fields.

### Phase 2: Language Code Service Bootstrap

Prerequisite for Phase 3. Steps:

1. Deploy the Language Code Service.
2. Create the `EPMS` code source record.
3. Load all EPMS language codes via the import process:
   - Submit import job with EPMS code payload.
   - Compliance reviewer reviews and approves the import.
   - Import is applied; `snapshot.imported` Kafka event fires.
4. Create the `ISO_639_3` code source record.
5. Load all ISO 639-3 language codes via the import process (same human-review flow).
6. Manually create EPMS ↔ ISO 639-3 mappings for all active EPMS codes via the API or import tool.
7. **Validation gate:** Confirm 100% of active EPMS language codes have at least one mapping to ISO 639-3 before proceeding to Phase 3.

**Note on ISO 639-3 import volume:** ISO 639-3 contains ~7,916 entries. While human sign-off is required per compliance policy, individual-code review is not practical at this scale. The recommended bulk review process is:
   - Reviewer downloads the import payload (SIL machine-readable file) and performs a documented spot-check of ≥50 randomly selected codes against the SIL public registry.
   - Reviewer documents the spot-check results (codes verified, any discrepancies found) in the `import_job.review_notes` field.
   - Reviewer approves the import via the Admin UI (or API), recording their identity and timestamp.
   - The `import_job` record (with `reviewed_by`, `reviewed_at`, `review_notes`) constitutes the compliance audit record.
   - **This process must be agreed and approved by the compliance team before Phase 2 begins.** If the compliance team requires a different review threshold or process, update this section accordingly.

### Phase 3: Preference Schema Extension + Language Code Transformation

Once Phase 2 is complete and all mappings are validated:

1. **Schema migration:** Add `written_language_legacy_epms_code`, `spoken_language_legacy_epms_code` fields to the CPMS preference record. These are nullable initially.

2. **Transformation batch job** (idempotent, resumable):
   - For each preference record in CPMS:
     a. Copy `written_language_code` → `written_language_legacy_epms_code`
     b. Copy `spoken_language_code` → `spoken_language_legacy_epms_code`
     c. Call Language Code Service `POST /translate` for written language: EPMS → ISO 639-3
     d. Call Language Code Service `POST /translate` for spoken language: EPMS → ISO 639-3
     e. Update `written_language_code` and `spoken_language_code` with ISO 639-3 codes
   - Records that fail translation (unmapped code) are handled as follows:
     - The legacy EPMS code is still copied to `*_legacy_epms_code` (copy step is unconditional).
     - The ISO code fields (`written_language_iso_code`, `spoken_language_iso_code`) are left **null** for that preference.
     - The record is written to an **unmapped-codes error log** (preference ID, EPMS code, failure reason) for SME review.
     - The preference record is otherwise complete and valid — a null ISO code field is a known, tracked exception, not a broken record.
     - A post-job reconciliation report must show: total records processed, total mapped, total unmapped, and the full unmapped-codes log. **Phase 3 is not considered complete until unmapped count reaches zero** (all exceptions resolved by adding missing mappings and re-running).
   - Job is idempotent: skip records where `written_language_legacy_epms_code` is already populated AND ISO field is non-null. Records with legacy field populated but null ISO field are retried (they may have been unmapped on a previous run and now have a mapping).

3. **CPMS API updates:** Update all CPMS preference APIs, Kafka messages, and file outputs to:
   - Include `written_language_iso_code`, `spoken_language_iso_code` (ISO 639-3 values)
   - Include `written_language_legacy_epms_code`, `spoken_language_legacy_epms_code`
   - Existing fields (`written_language_code`, `spoken_language_code`) now carry ISO 639-3 values

### Phase 4: EPMS Legacy API Compatibility Layer

Deploy the pass-through proxy services. These are thin transformation layers — they do not own any state.

**Request flow (inbound EPMS call → CPMS):**
1. EPMS-format request arrives (with EPMS language code, e.g., `HUM-ENG-01`)
2. Compatibility layer transforms EPMS schema → CPMS schema
3. Language code field: pass EPMS code as-is (CPMS will store it in the legacy field)
4. Forward transformed request to CPMS API
5. Transform CPMS response → EPMS format
6. Return to caller

**Response flow (outbound to EPMS callers):**
- Language code fields in the response use the stored `legacy_epms_code` from the preference (authoritative source for legacy consumers)
- If `legacy_epms_code` is null (e.g., for preferences created natively in CPMS without an EPMS origin), the compatibility layer calls `POST /translate` on the Language Code Service to reverse-map ISO 639-3 → EPMS.
  - **Reverse-translate fallback contract:** If no reverse mapping exists for a CPMS-native ISO code:
    - The legacy API response returns the field as **null** (not the ISO code, not an error).
    - The response HTTP status remains 200 — the preference is valid; the language code is simply unavailable in EPMS representation.
    - Callers of the legacy EPMS API that require a non-null language code must handle the null case explicitly.
    - The compatibility layer logs each null reverse-translation (preference ID, ISO code) for monitoring and future mapping gap resolution.
  - This scenario is expected to be rare if the EPMS-to-CPMS migration is complete, since all migrated preferences will have `legacy_epms_code` populated. CPMS-native preferences (created post-migration via CPMS APIs) are the only case where reverse translation is needed.

**Preserved EPMS interface surface:**
- All existing EPMS REST API endpoints
- All existing EPMS Kafka topic formats
- All existing EPMS file formats (batch/ETL)

No breaking changes to EPMS-consuming systems.

## 10. Code Source Extensibility

The data model is designed to support additional language code sources beyond ISO 639-3 and EPMS without schema changes. To add a new code source (e.g., BCP 47/IETF, HL7 FHIR ValueSet, regional code sets):

1. Create a new `code_source` record.
2. Load codes via the standard import process.
3. Create mappings between the new source and existing sources (ISO 639-3 is the recommended hub).
4. Consuming services automatically see the new source via Kafka events and REST API.

No migrations, no new tables, no code changes to the Language Code Service itself.

## 11. Caching & Performance

### Language Code Service
- Internal DB queries are read-heavy (lookups and translations); writes are infrequent (admin operations only).
- Add DB read replicas for lookup queries under load.
- Consider an in-process cache (Caffeine/Guava) for the translation endpoint with a short TTL (5 min) to reduce DB load during high-throughput migration jobs.

### Migration Job
- The Phase 3 transformation job should use the **batch translation endpoint** (`POST /translate/batch`) rather than individual calls to minimize network overhead.
- Process preferences in configurable batch sizes (e.g., 500 records per batch).
- Run with concurrency controls to avoid overwhelming the Language Code Service or the Preference DB.

### Legacy Compatibility Layer
- For preferences with `legacy_epms_code` already populated, no Language Code Service call is needed on outbound responses — serve directly from the stored field.
- Only preferences created natively in CPMS (no EPMS origin) require a reverse translation call.

## 13. Open Questions

1. **Existing CPMS consumers of language code fields (Phase 3 blocker):** Which downstream consumers (services, files, APIs) currently read `written_language_code` and `spoken_language_code`? These consumers will see the field value change from EPMS codes to ISO 639-3 codes post-Phase 3. **This is a hard blocker for Phase 3:** all consumers must be identified, notified, and updated (or confirmed to be tolerant of ISO codes) before the Phase 3 transformation job runs against production data. Required deliverable: a complete consumer inventory with readiness status for each, signed off before Phase 3 begins.

2. **EPMS code coverage:** Are all EPMS language codes mappable to ISO 639-3? Are there EPMS codes that represent regional dialects or proprietary categories with no ISO 639-3 equivalent? The validation gate in Phase 2 will surface this, but a preliminary audit before Phase 2 would reduce risk.

3. **ISO 639-3 bulk review process:** With ~7,916 ISO codes, what is the compliance team's accepted process for bulk sign-off? (e.g., a single reviewer approval for a SIL-sourced snapshot, with documented spot-check criteria?) This needs to be agreed before Phase 2 begins.

4. **Preference record volume:** How many CPMS preference records will the Phase 3 transformation job process? This determines batch job runtime estimates and capacity planning for the Language Code Service during migration.

5. **Macrolanguage handling:** ISO 639-3 distinguishes individual languages from macrolanguages (e.g., `ara` for Arabic as a macrolanguage, `arb` for Modern Standard Arabic as an individual language). Does CPMS need to distinguish at this level, or is macrolanguage precision sufficient?

6. **Language Code Service hosting:** What is the target deployment pattern (ECS/EKS/container)? Does it follow existing CPMS microservice deployment conventions?

7. **Authentication on Language Code Service:** Does the service require fine-grained authorization (e.g., only admin roles can call `POST /codes`)? Should it use the existing CPMS auth/authz patterns?

## 15. Approaches Considered

### Approach A: REST-First Language Code Service (Minimal Viable)
- REST API only; synchronous lookups; in-service cache; no Kafka events
- Completeness: 6/10 — misses audit trail, adds latency to migration job, no change notifications
- Viable as an interim phase if timeline is aggressive

### Approach B: Event-Driven Reference Data Service (Recommended — this design)
- REST API + Kafka events; consumer-side caching; full code/mapping lifecycle events
- Completeness: 9/10 — covers performance, compliance audit trail, change propagation
- Higher initial effort; justified by compliance requirements and migration job scale

### Approach C: Graph-Traversal Mapping Engine
- Directed graph for multi-hop translations; handles indirect code system relationships
- Completeness: 8/10 for future extensibility, but 5/10 for current needs
- Deferred — revisit if CPMS adds BCP 47, HL7 FHIR, or other code systems requiring multi-hop resolution

---

## 16. Recommended Approach

**Approach B — Event-Driven Reference Data Service**, for three reasons:

1. **Migration job performance:** The Phase 3 transformation job will process potentially millions of preference records. Synchronous service-call-per-record would be a throughput bottleneck. Consumer-side caching via Kafka eliminates this bottleneck entirely.

2. **Compliance audit trail:** An immutable event log (every code addition, deprecation, and mapping change) satisfies healthcare compliance requirements for "who changed what when" without bolt-on audit logging.

3. **Infrastructure leverage:** CPMS already runs Kafka. Adding new topics for language code events is marginal cost, not new infrastructure.

## What I noticed about how you think

- You framed the problem from the migration sequence outward — "what must be true at each phase before the next phase can proceed." That's systems thinking, not feature thinking. It surfaces dependencies early and makes the design testable at each gate.

- You specified "other language code sources that might need to be supported" before anyone asked. That's the right instinct: don't hardcode a two-system assumption into a schema that will outlive the migration.

- You identified the proxy/pass-through pattern for legacy API compatibility without being prompted. That's the correct architecture (strangler fig), and naming it explicitly up front means the legacy compatibility layer stays thin and purposeful rather than accumulating logic over time.

- You said "CPMS will need a trusted copy of EPMS language codes." The word "trusted" is doing a lot of work there — it implies governance, not just a data dump. That framing is what leads to the import review workflow instead of a raw table copy.
